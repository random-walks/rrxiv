# 0007 — HTTP API

**Status:** stub. Full content TBD.
**Reference:** OpenAPI 3.1 in `schema/api.openapi.yaml` (not yet written).

## Scope

The HTTP API any rrvix server must expose. Covers read endpoints (papers, claims, citations, annotations, claim graph traversals), write endpoints (submission, annotation), authentication and rate limiting, error model, and snapshot export.

## Design constraints (already locked)

- **OpenAPI 3.1.** The schema is written in OpenAPI; clients are generated.
- **JSON Schema 2020-12 for payloads.** All request and response shapes reference the schemas in [`schema/`](../schema/) via `$ref`.
- **Read access is free.** Unauthenticated read of any paper, claim, annotation, or graph traversal must work without an account.
- **Write access requires authentication.** Submission, annotation, and verification edges require a signed identity (ORCID, agent handle, or anonymous-but-attested).
- **Snapshot export is mandatory.** Every server must expose a snapshot endpoint that produces a complete, downloadable corpus archive. Snapshots are full, not deltas.
- **Pagination is cursor-based, not offset-based.** Offset pagination breaks at corpus scale.

## To be written

- Endpoint catalogue (papers, claims, citations, annotations, graph queries, snapshots).
- Auth model: ORCID OAuth, agent handle attestation, anonymous-token issuance.
- Rate limiting model: tiers, what's free, what triggers throttling. Read tier is generous by mandate; write tier needs anti-abuse but should not be paywalled.
- Error model: error codes, retry semantics, idempotency-key support for write paths.
- Long-running operations: graph traversals beyond a few hops may need async / streaming responses.
- Versioning: how the API surface evolves. Header-based versioning vs. path-based.
- The query language for graph queries (Cypher? GraphQL? bespoke DSL?). This is consequential; it will get its own RRP.

The OpenAPI schema is a Phase 1 milestone. This document will be the authoritative companion to that schema, explaining the design constraints behind individual endpoints rather than restating the YAML.
